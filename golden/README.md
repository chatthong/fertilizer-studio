# Golden vectors — calculation engine

Reference input→output fixtures captured from the **original (pre-rewrite) TypeScript
calculation engine**. The Rust port (Phase 2) must reproduce every output here within a
tight tolerance. All numbers are recorded at full f64 precision.

These exist because the legacy solver is a hand-rolled NNLS with idiosyncratic
tie-breaking that no off-the-shelf crate will reproduce — the only safe way to port it is
to match known-good outputs exactly.

## Files

| File | Function | Coverage |
|------|----------|----------|
| `blend-optimizer.json` | `optimizeBlendShares` (NNLS blend solver) | 14 cases: single/multi-product blends, oxide targets, weighted targets, max-constraint **bisection**, liquid basis, null/empty edges, **zero-target weight-500 buffer pattern** (the common real call), liquid+constraint, non-binding constraint, oxide-key constraint, 5-product active-set stress |
| `oxide-to-element.json` | `convertOxidePercentToElement` | P₂O₅→P (×0.4364602764), K₂O→K (×0.8301472547), no-factor keys, 0/null |
| `percent-to-ppm.json` | `convertPercentValueToPpm` | ppm = %/100 × g/L × 1000, incl. 0/null/zero-g/L |
| `percent-record-to-ppm.json` | `convertPercentToPpm` | record variant |

## Constants (must match in the Rust port)

`P2O5_TO_P = 0.4364602764` · `K2O_TO_K = 0.8301472547` · `EPSILON = 1e-9` ·
`DEFAULT_LAMBDA = 1e-6` · `LIQUID_SCALING_FACTOR = 10000` · solver tolerance `1e-8`,
constraint tolerance `1e-3`.

## How the Rust tests use these

Load each JSON, run the ported function on `input`, assert `output` matches:
- **conversions** — exact (or within 1e-12).
- **solver** — within ~1e-6 absolute on each usage/predicted value.

⚠️ Quirk to preserve: `predictedPpmByNutrient` reports **element-basis** ppm for oxide keys
(e.g. a K₂O target of 200 predicts `166.03` = 200 × 0.8301). The port must match this.

## Provenance & regeneration

Generated by a harness that runs the original TS **verbatim** (only `@/…` import paths
rewritten to relative). The harness lives in the private legacy repo at
`tools/golden-capture/` (`capture.mts` + copies of `target-optimizer` / `ppm-converter`),
run with `node capture.mts` on Node ≥ 23 (native TS type-stripping).
